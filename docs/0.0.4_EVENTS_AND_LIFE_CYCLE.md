# STABILIZATION 0.0.4: Events and Lifecycle

This document defines the `0.0.4` stabilization line, focused on the event model and public lifecycle hooks across the framework core.

Scope of this stabilization:
- establish a small, explicit event system shared by core components
- expose lifecycle hooks at stable execution boundaries already present in the framework
- support both per-instance listeners and class-level listeners without introducing a new execution model
- cover the public lifecycle hooks with automated tests
- preserve the stabilized contracts from `0.0.1`, `0.0.2` and `0.0.3`

Out of scope for now:
- asynchronous event delivery
- listener priority, propagation stop, or wildcard subscriptions
- replacing the controller/router/request execution model
- broad plugin or extension APIs beyond the lifecycle hooks documented here

- listeners remain synchronous and in-process
- instance listeners run on the emitting object

Notes:
- `Observable` now exposes:
  - `on(string $event, callable $handler): void`
  - `trigger(string $event, ...$args): void`
- instance listeners are stored on the emitting object
- `trigger()` always passes the emitter as the first listener argument
- `Event` now exposes:
  - `Event::on(string $class, string $event, callable $handler): void`
  - `Event::trigger(string $class, string $event, mixed $emitter, ...$args): void`
  - `Event::clear(?string $class = null, ?string $event = null): void`
- class-level dispatch resolves:
  - the concrete emitted class
  - parent classes
  - implemented interfaces
- this allows listeners registered on a base type such as `BaseRequest::class` or `BaseResponse::class` to observe subclass lifecycle events

Acceptance criteria:
- any observable framework object may support both local and global listeners through a single event model

## Short Public Event Contract

This section consolidates the short public event contract stabilized by `0.0.4`.

### Main public surface

- `Adige\core\events\Observable`
- `Adige\core\events\Event`
- lifecycle event constants documented on:
  - `BaseRequest`
  - `BaseResponse`
  - `App`
  - `ExceptionHandler`
  - `ActiveRecord`

### Listener registration

- per-instance listeners:
  - `$object->on($event, $handler)`
- class-level listeners:
  - `Event::on(SomeClass::class, $event, $handler)`

### Dispatch behavior

- the emitter object is always the first listener argument
- extra trigger arguments are forwarded in order
- instance listeners run from the emitting object
- class-level listeners registered on:
  - the concrete class
  - any parent class
  - any implemented interface
  receive the emitted event

### Cleanup behavior

- `Event::clear()` removes all global listeners
- `Event::clear(SomeClass::class)` removes all listeners for a class
- `Event::clear(SomeClass::class, $event)` removes listeners only for that class/event pair

### Limits of the current public contract

- listeners are synchronous
- listener priority ordering is not a documented public feature
- stopping propagation is not part of the public contract
- queued, deferred, or distributed event delivery is out of scope
- the existence of lifecycle hooks is public; every internal formatting or helper detail around them is not

## Immediate outcome

Main outcome of `0.0.4`:
- the framework now has a documented and tested event surface aligned with its stabilized lifecycle boundaries
- applications can observe request, response, error, app and model transitions without coupling to unstable internals
