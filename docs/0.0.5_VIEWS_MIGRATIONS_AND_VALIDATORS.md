# STABILIZATION 0.0.5: Views, Migrations and Validators

This document defines the `0.0.5` stabilization line, focused on the new view contract, the first migration workflow, and the first stable validator contract around `ActiveRecord`.

Scope of this stabilization:
- stabilize a small synchronous view layer integrated with the current controller/app flow
- define the first public migration contract around file discovery, execution and metadata tracking
- define the first built-in validator set integrated with `ActiveRecord::rules()` and `save()`
- keep bootstrap configuration explicit for view and migration paths
- preserve the stabilized contracts from `0.0.1` through `0.0.4`

Out of scope for now:
- a full template engine beyond PHP view files
- layouts, sections or asset pipelines as part of the stable contract
- advanced migration operations such as rename table, rename column or cross-database schema diffing
- database-backed relational validators such as `exist`
- transport-specific validators such as `file`, `image` or `captcha`

## Phase 2: Stabilize the first migration contract

Goal:
- provide a predictable schema-change workflow aligned with the current ORM architecture

Current status:
- completed

Tasks:
- [x] add a public `Migration` base abstraction
- [x] add a fluent `MigrationField` builder
- [x] add explicit dialect compilation for MySQL and SQLite
- [x] support `createTable`, `dropTable`, `addColumn` and `dropField`
- [x] support explicit table indexes in `createTable`
- [x] guarantee the metadata table exists before migration execution
- [x] make migration discovery depend on filename instead of class name
- [x] allow anonymous-class migrations returned by file

Notes:
- migration files now follow the stabilized format:

```php
<?php

use Adige\core\database\Migration;

return new class extends Migration {
    public function up(): void
    {
    }

    public function down(): void
    {
    }
};
```

- `Migration` now supports connection injection after construction through `setConnection()`
- `Migration` now supports explicit table indexes through `index(array|string $columns, ?string $name = null)`
- migration metadata lives in the `migrations` table with:
  - `id`
  - `name`
  - `batch`
  - `created_at`
- metadata creation/upgrade now also ensures the `batch` column exists for older metadata tables
- `createTable()` now accepts both `MigrationField` and `MigrationIndex` entries in the same definition array

Example:

```php
$this->createTable('posts', [
    $this->field('id')->integer()->autoIncrement(),
    $this->field('title')->string(120)->notNull(),
    $this->field('published')->boolean()->default(false),
    $this->index('title'),
    $this->index(['title', 'published'], 'posts_title_published_unique')->unique(),
]);
```

Acceptance criteria:
- migration execution is deterministic across supported drivers and tied to a small documented contract

## Short Public Contract

### Views

- `BaseView` is the stable base view component
- controller rendering uses `BaseController::render()`
- view resolution supports:
  - a configured base directory
  - explicit aliases such as `@shared/partial`
- view names reject path traversal and absolute paths
- render hooks are part of the public lifecycle contract

### Migrations

- migration identity is the filename
- a migration file must return a `Migration` instance
- the migration object may receive its `Connection` after construction
- `createTable()` accepts fields and indexes in a single definition array
- `migrate/up` applies all pending migrations in one batch
- `migrate/down` reverts the latest batch, or the latest `N` batches when `steps` is provided
- metadata is tracked in the `migrations` table through `name`, `batch` and `created_at`

### Validators

- model validation is part of the stable `ActiveRecord` contract in this line
- rules are declared through `rules(): array`
- `save()` validates by default and accepts an explicit skip-validation flag
- built-in validators cover required presence, scalar types, ranges, patterns, date/url/email formats and uniqueness
