# 1.0.0-alpha Checkout

Este documento fecha o recorte de `1.0.0-alpha` com base no estado atual do pacote, no `PUBLIC_API.md` e no checklist concluído em `TODO.md`.

Objetivo deste checkout:
- registrar o que efetivamente entrou no corte `1.0.0-alpha`
- deixar explícito o que já pode ser tratado como contrato utilizável
- separar o que está pronto para release do que foi intencionalmente adiado

## Status do corte

Estado atual:
- `1.0.0-alpha` tecnicamente pronto para empacotamento e tag

Resumo executivo:
- o pacote deixou de depender da pasta `/app`
- o fluxo de consumo via Composer foi estabilizado
- o entrypoint oficial de console passou a ser `vendor/bin/adige`
- `views`, `migrations` e `validators` já compõem a baseline pública do framework
- existe cobertura de contrato para consumo como pacote por uma aplicação externa

## O que foi consolidado neste alpha

### 1. Distribuição como pacote Composer

O pacote agora assume explicitamente o modelo de biblioteca:
- `composer.json` está alinhado para distribuição como `library`
- o binário oficial do pacote é `bin/adige`
- o comando exposto ao consumidor é `vendor/bin/adige`
- o launcher opcional `./adige` pode ser criado via `install/index`

Consequência prática:
- o framework não depende mais de execução a partir da raiz do próprio repositório
- a aplicação consumidora define seu próprio `basePath`

### 2. Remoção das convenções implícitas antigas

O pacote não trata mais `/app` como fronteira ativa do projeto.

Foram consolidadas as seguintes decisões:
- bootstrap da aplicação vem do `basePath` do projeto consumidor
- autodiscovery de controllers não infere namespace magicamente da raiz
- autodiscovery agora exige `controllerNamespaces` explícito
- migrations usam por padrão `<basePath>/migrations`

### 3. Core público alinhado para consumo externo

O contrato público do entrypoint ficou explícito em torno de:
- `Adige::run(?string $appClass = null, ?string $basePath = null): void`
- `Adige::setBasePath(string $basePath): void`
- `Adige::basePath(): string`
- `Adige::packageRoot(): string`
- `Adige::vendorDir(): string`

Isso fecha a separação entre:
- raiz do pacote
- diretório `vendor`
- raiz lógica da aplicação consumidora

### 4. Views estabilizadas

O sistema de views agora tem um contrato pequeno e previsível:
- render síncrono por arquivos PHP
- diretório de views por instância
- aliases explícitos
- proteção contra path traversal e paths absolutos
- limpeza correta de buffer em falha
- hooks de lifecycle de render

Esse recorte já está documentado e coberto como parte da baseline `0.0.5`.

### 5. Migrations estabilizadas

O primeiro fluxo estável de migrations foi fechado com:
- discovery por filename
- suporte a migration retornando classe anônima
- criação automática da tabela `migrations`
- suporte a `batch`
- `up` executando pendências em um único batch
- `down` revertendo batches mais recentes
- suporte a `createTable`, `dropTable`, `addColumn`, `dropField`
- suporte a índices na criação de tabela

Esse comportamento já está alinhado com o console controller de migrations.

### 6. Validators estabilizados

`ActiveRecord` agora possui fluxo de validação integrado ao `save()`, com possibilidade de skip explícito.

Baseline atual:
- rules no formato inspirado em Yii2
- built-ins para regras frequentes
- suporte a validadores customizados por classe
- `unique` com suporte a `targetAttribute` composto

Built-ins já empacotados neste corte incluem:
- `required`
- `boolean`
- `integer`
- `number`
- `string`
- `minLength`
- `maxLength`
- `in`
- `compare`
- `date`
- `url`
- `mask`
- `email`
- `unique`

## Cobertura e verificação

Checklist técnico já concluído:
- `composer validate --strict --no-check-publish` limpo
- fixture real de aplicação consumidora criada em `tests/Fixtures/consumer-app`
- cobertura contratual para web, console, views e migrations nesse cenário de consumo
- suíte unitária ajustada para PHPUnit 12 sem os notices anteriores

Último resultado validado da suíte:
- `OK (145 tests, 572 assertions)`

## O que fica fora deste corte

Itens deliberadamente adiados:
- homepage pública do projeto, possivelmente via GitHub Pages
- superfícies maiores de template engine além de views PHP
- operações avançadas de migration como rename table e rename column
- validadores mais específicos de upload, imagem, captcha e cenários relacionais como `exist`

Esses itens não bloqueiam o `1.0.0-alpha`.

## Leitura final do release

O `1.0.0-alpha` já pode ser tratado como um primeiro corte distribuível do Adige com:
- core HTTP e console coerentes
- ORM utilizável
- views utilizáveis
- migrations utilizáveis
- validators utilizáveis
- consumo externo via Composer documentado

O que resta para publicar este corte é operacional:
- criar a tag `1.0.0-alpha`
- publicar o release no momento adequado
