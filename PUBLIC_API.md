# Adige Public API

This document consolidates Adige's public baseline for `1.0.0-alpha`.

Historical line that led to this cut:
- `0.0.1` core stabilization
- `0.0.2` ORM stabilization
- `0.0.3` legacy console cleanup
- `0.0.4` events and lifecycle
- `0.0.5` views, migrations and validators

Objective:
- make explicit what applications can use with a reasonable expectation of compatibility
- separate public API from internal detail
- consolidate the stable microframework contract in a single place

## Current Positioning

In its current state, Adige can be treated as a small microframework with:
- a stabilized HTTP/kernel/router core
- a stabilized ORM in the `ActiveRecord` / `Connection` / `Schema` slice
- a console flow consolidated around conventional controller/route behavior
- distribution as a Composer `library`
- a public baseline closed for `1.0.0-alpha`

Recommended use:
- simple APIs
- internal tools
- small or controlled applications
- scenarios where predictability and low coupling matter more than large feature scope

Current distribution model:
- the package is consumed via Composer
- the official console entrypoint is `vendor/bin/adige`
- the consuming project defines its own `basePath`
- the optional `./adige` launcher can be generated by `install/index`

## Consolidated Architectural Decision

- `Adige` is the central entrypoint of the execution cycle
- `App` is the configurator/container responsible for handlers, bootstrap and response normalization
- `Router` resolves explicit routes and convention-based autodiscovery
- controllers return simple results and `App` normalizes them into `BaseResponse`
- the console uses the same controller/action model as the web flow

## Core Public API

### `Adige::run()`

File:
- [src/core/Adige.php](/home/mathmpr/PhpstormProjects/adige/src/core/Adige.php)

Contract:
- initializes the common environment
- instantiates the configured application
- executes the main cycle
- handles exceptions through `ExceptionHandler`
- accepts an explicit `basePath` for the consuming project

Expected usage:

```php
use Adige\core\Adige;

Adige::run();
Adige::run(null, __DIR__);
```

Guarantees:
- `Adige::run()` remains the main entrypoint
- the cycle goes through request, router, action and response
- the final transport is normalized by `App`
- when provided, `basePath` defines the consuming project's logical root
- when omitted, the framework resolves `basePath` from `APP_ROOT`, `BASE_PATH`, `ADIGE_BASE_PATH`, `vendor/` and `getcwd()`

Stabilized public behaviors:
- `Adige::run(?string $appClass = null, ?string $basePath = null): void`
- `Adige::setBasePath(string $basePath): void`
- `Adige::basePath(): string`
- `Adige::packageRoot(): string`
- `Adige::vendorDir(): string`

### `Route::*()`

File:
- [src/core/routing/Route.php](/home/mathmpr/PhpstormProjects/adige/src/core/routing/Route.php)

Public API:
- `Route::get()`
- `Route::post()`
- `Route::put()`
- `Route::delete()`
- `Route::patch()`
- `Route::options()`
- `Route::head()`
- `Route::all()`
- `Route::group()`

Contract:
- register explicit routes
- accept either a `callable` or controller/action target
- `group()` applies group prefix and middleware

Guarantees:
- explicit routes take precedence over autodiscovery
- among compatible explicit routes, static segments take precedence over dynamic ones
- `ALL` matches any method

### Controllers

Base file:
- [src/core/controller/BaseController.php](/home/mathmpr/PhpstormProjects/adige/src/core/controller/BaseController.php)

Contract:
- controllers expose `action*` actions
- `beforeAction()` and `afterAction()` participate in the cycle
- `render()` delegates to the view component configured in the app
- actions may return:
  - `BaseResponse`
  - `string`
  - `array`
  - `object`
  - `null`

Guarantees:
- action return values are always normalized into `BaseResponse`
- route params take precedence in action parameter injection
- the same action/return model applies to both web and console
- when a `view` component is configured, controllers can render views by logical name

### Views

Base file:
- [src/core/BaseView.php](/home/mathmpr/PhpstormProjects/adige/src/core/BaseView.php)

Related configuration:
- `Adige::VIEW_HANDLER`

Contract:
- `BaseView` is the public component for synchronous PHP-file rendering
- default resolution uses a base directory configured per instance
- explicit aliases can be registered, for example `@shared/partial`
- the render signature remains:
  - `render(string $view, array $params = []): string`

Stabilized public behaviors:
- `escape()` exposes basic HTML escaping
- `setViewDirectory()` and `getViewDirectory()` control the base directory
- `registerAlias()` and `getAliases()` control explicit aliases
- view parameters are exposed through `extract(..., EXTR_SKIP)`

Guarantees:
- the view directory is instance state, not class-global state
- consecutive renders do not leak resolved paths between calls
- view names reject path traversal and absolute paths
- failures during render correctly clean the output buffer

Public lifecycle events:
- `BaseView::EVENT_BEFORE_RENDER`
- `BaseView::EVENT_AFTER_RENDER`
- `BaseView::EVENT_RENDER_ERROR`

Expected usage:
- applications configure the `view` handler in app/bootstrap
- controllers use `render()` to delegate composition to `BaseView`
- aliases are explicit and do not depend on magic project-root conventions
- the consuming application's bootstrap lives under `basePath`, not at the package root

### App lifecycle

File:
- [src/core/App.php](/home/mathmpr/PhpstormProjects/adige/src/core/App.php)

External contract:
- `App` remains the response normalization and final transport emission point
- `App` remains the lazy resolution point for configured handlers, including `view` and `migrations`
- bootstrap is loaded from the consuming project's `basePath`

Public lifecycle events:
- `App::EVENT_BEFORE_NORMALIZE_RESPONSE`
- `App::EVENT_AFTER_NORMALIZE_RESPONSE`
- `App::EVENT_BEFORE_EMIT_RESPONSE`
- `App::EVENT_AFTER_EMIT_RESPONSE`

Expected usage:
- listeners can observe the raw action result before normalization
- listeners can observe the final `BaseResponse` before and after app dispatch
- this is useful for logs, tracing, metrics and cross-cutting integration without coupling into the kernel
- applications can provide explicit configuration for:
  - `Adige::VIEW_HANDLER`
  - `Adige::MIGRATIONS_CONFIG`
  - `Adige::ROUTER_HANDLER`
  - `Adige::REQUEST_HANDLER`
  - `Adige::RESPONSE_HANDLER`

## ORM Public API

### `Connection`

File:
- [src/core/database/Connection.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/Connection.php)

Contract:
- represents a named connection with explicit driver selection
- supports MySQL and SQLite without silent fallback

Stabilized public behaviors:
- `getDefaultConnection()` returns the default connection or fails explicitly
- duplicate connections by name fail explicitly
- `query()` returns `PDOStatement`
- `select()` returns `array`
- `insert()` returns `lastInsertId()`
- `update()` returns affected rows
- `delete()` returns affected rows

Stabilized policy:
- the first connection may become default explicitly
- later connections do not implicitly steal default status
- connection errors are wrapped by `CantConnectException`
- query errors surface as `PDOException`

### `Schema`

### `ActiveRecord`

File:
- [src/core/database/ActiveRecord.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/ActiveRecord.php)

Contract:
- the ORM's main model continues to be `ActiveRecord`
- explicit connections may be passed through query and persistence flows

Stabilized public behaviors:
- runtime connection flows through `one()`, `all()`, `build()`, `execute()`, `save()` and `remove()`
- hydrated models preserve the connection used by the main query
- relations reuse the same connection in eager and lazy loading
- `where()` uses recursive logical groups and explicit operators
- eager loading supports nested paths via dot notation
- `save()` validates the model before persistence by default
- `save($connection, true)` allows explicit validation skipping
- `validate()` executes the rules declared on the model
- validation errors are available through `addError()`, `getErrors()`, `clearErrors()` and `hasErrors()`

Public lifecycle events:
- `ActiveRecord::EVENT_BEFORE_INSERT`
- `ActiveRecord::EVENT_AFTER_INSERT`
- `ActiveRecord::EVENT_BEFORE_UPDATE`
- `ActiveRecord::EVENT_AFTER_UPDATE`
- `ActiveRecord::EVENT_BEFORE_DELETE`
- `ActiveRecord::EVENT_AFTER_DELETE`
- `ActiveRecord::EVENT_BEFORE_LOAD`
- `ActiveRecord::EVENT_AFTER_LOAD`
- `ActiveRecord::EVENT_BEFORE_HYDRATE`
- `ActiveRecord::EVENT_AFTER_HYDRATE`

Expected usage:
- listeners can be registered per instance with `on()`
- global listeners can be registered by base or concrete class through [src/core/events/Event.php](/home/mathmpr/PhpstormProjects/adige/src/core/events/Event.php)
- this allows validation, auditing, state transformation and side effects around the model lifecycle

### `ActiveRecord` validators

Files:
- [src/core/database/validators/ValidatorInterface.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/validators/ValidatorInterface.php)

Contract:
- models may declare `rules(): array`
- each rule follows the general format:
  - `[['field'], 'validator']`
  - `[['field'], 'validator', 'namedParam' => 'value']`
  - `[['field'], CustomValidator::class, ...$args]`
- built-in and custom validators share the same flow through `ValidatorInterface`

Stabilized public built-ins:
- `required`
- `boolean`
- `integer`
- `number`
- `string`
- `minLength`
- `maxLength`
- `in`
- `compare`
- `date`
- `url`
- `mask`
- `email`
- `unique`

Supported public aliases:
- `bool`
- `int`
- `float`
- `double`
- `decimal`
- `min_length`
- `max_length`
- `range`
- `match`
- `regex`
- `regexp`
- `datetime`

Guarantees:
- validators receive the model, the fields, normalized params and the optional connection
- `unique` supports simple and composite uniqueness
- the composite `targetAttribute` format is part of the supported contract, for example:

```php
[['date', 'app'], 'unique', 'targetAttribute' => ['date', 'app'], 'message' => 'The combination of Date and App has already been taken.']
```

Expected usage:
- model-local business rules should live in `rules()`
- applications may provide custom validators by class as long as they implement `ValidatorInterface`

### `Migration`

Files:
- [src/core/database/Migration.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/Migration.php)
- [src/core/database/MigrationField.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/MigrationField.php)
- [src/core/database/MigrationIndex.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/MigrationIndex.php)
- [src/core/database/MigrationDialect.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/MigrationDialect.php)

Public support dialects:
- [src/core/database/dialects/mysql/MysqlMigration.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/dialects/mysql/MysqlMigration.php)
- [src/core/database/dialects/sqlite/SqliteMigration.php](/home/mathmpr/PhpstormProjects/adige/src/core/database/dialects/sqlite/SqliteMigration.php)

Contract:
- migrations are files discovered by filename
- the filename is the canonical migration identity
- a migration file must return a `Migration` instance
- the connection can be injected after construction via `setConnection()`

Stabilized public behaviors:
- `createTable()`
- `dropTable()`
- `addColumn()`
- `dropField()`
- `field()` to create `MigrationField`
- `index()` to create `MigrationIndex`
- `executeUp()`
- `executeDown()`
- `MigrationField` exposes a fluent builder with:
  - `type()`
  - `integer()`
  - `string()`
  - `text()`
  - `boolean()`
  - `timestamp()`
  - `nullable()`
  - `notNull()`
  - `default()`
  - `primary()`
  - `unique()`
  - `autoIncrement()`
- `MigrationIndex` exposes a fluent builder with:
  - `columns()`
  - `name()`
  - `unique()`

Guarantees:
- the `migrations` metadata table is guaranteed before execution
- metadata includes:
  - `name`
  - `batch`
  - `created_at`
- `executeUp()` runs pending migrations inside the framework transaction flow
- `executeDown()` reverts through the same flow
- the framework supports MySQL and SQLite without silent fallback for migrations as well
- `createTable()` accepts both `MigrationField` and `MigrationIndex` in the same definition array
- simple and composite indexes are compiled explicitly for MySQL and SQLite

Expected usage:
- new migrations use the format:

```php
use Adige\core\database\Migration;

return new class extends Migration {
    public function up(): void
    {
    }

    public function down(): void
    {
    }
};
```

- applications can configure the discovery path through `Adige::MIGRATIONS_CONFIG`
- migrations may define indexes, for example:

```php
$this->createTable('posts', [
    $this->field('id')->integer()->autoIncrement(),
    $this->field('title')->string(120)->notNull(),
    $this->index('title'),
    $this->index(['title', 'published'], 'posts_title_published_unique')->unique(),
]);
```

## Known Limits and Residuals

Items still present, but which do not invalidate the stabilized baseline:

1. `Adige::$app` still exists as a relevant global point.
2. Autodiscovery remains in the router core and keeps some coupling to application conventions.
3. The repository is still consolidating the transition from local playground to pure Composer package.
4. The public contract is stabilized at the current cut, but that does not imply broad feature scope.

## Expected Compatibility

The intention is to preserve:
- `Adige::run()`
- `Adige::run(..., $basePath)` as a compatible entrypoint for consuming projects
- `Route::*()`
- action return contract
- basic `Request` and `Response` contract
- documented routing/autodiscovery behavior
- public `ActiveRecord`, `Connection` and `Schema` baseline
- current console flow based on controller/action
- usage of `vendor/bin/adige` as the official console entrypoint

The following remain freer to evolve:
- internal `App` details
- internal `ExceptionHandler` implementation
- ORM dialect structure
- documentary leftovers and transition details from the old playground
